## Phase 1: x402 API Server

### Architecture
```
Agent → POST /api/consult (x402 payment: $20 USDC)
     → 402 Payment Required (first request)
     → Agent pays via x402 facilitator
     → Agent retries with payment proof
     → Server validates payment, creates job on-chain, runs consultation
     → Returns consultation result (build plan)
```

### Endpoints
| Endpoint | Price | Description |
|---|---|---|
| `POST /api/consult/quick` | $20 | Quick consultation — send project description, get build plan |
| `POST /api/consult/deep` | $30 | Deep consultation — detailed architecture review |
| `POST /api/audit` | $200 | Smart contract audit — send contract address or source |
| `POST /api/qa` | $50 | QA report — send dApp URL, get quality report |
| `GET /api/services` | Free | List available services and prices |
| `GET /api/job/:id` | Free | Check job status and results |

### Request/Response Format
```json
// POST /api/consult/quick
{
  "description": "I want to build a token dashboard for CLAWD",
  "context": "optional additional context, links, etc"
}

// Response (after payment)
{
  "jobId": 42,
  "status": "processing",
  "estimatedTime": "5-10 minutes",
  "resultUrl": "/api/job/42"
}

// GET /api/job/42 (when complete)
{
  "jobId": 42,
  "status": "completed",
  "result": {
    "buildPlan": "...",
    "gistUrl": "https://gist.github.com/...",
    "recommendedService": "BUILD_DAILY"
  }
}
```

### Tech Stack
- **Next.js API routes** (already in the project — `app/api/`)
- **@x402/next** `withX402` wrapper — gates API routes, settles after success
- **@x402/evm** `ExactEvmScheme` — EVM payment scheme for Base
- **Default facilitator** at `https://x402.org/facilitator` (Coinbase-hosted, free, no setup)
- **Worker assignment** — auto-assign to available worker bot
- **USDC goes directly to clawdbotatg.eth** — no on-chain contract interaction needed for payment

### x402 Payment Flow (v2 protocol)
1. Agent sends `POST /api/consult/quick` without payment
2. Server responds `402` with `PAYMENT-REQUIRED` header (base64 JSON with price, network, payTo)
3. Agent's x402 client signs USDC Permit2 approval + payment on Base
4. Agent retries with `PAYMENT-SIGNATURE` header (base64-encoded payment proof)
5. Server calls facilitator to verify → handler runs → facilitator settles USDC on Base
6. **Key:** Settlement happens AFTER handler succeeds (status < 400) — client never charged for failures
7. Server creates job, assigns worker, returns job ID
8. Worker bot processes the job
9. Result available at `/api/job/:id`

### Implementation Steps
1. `npm install @x402/core @x402/next @x402/evm`
2. Create shared x402 server config (`lib/x402.ts`)
3. Create API routes wrapped with `withX402` for each service
4. Build async job processing: create job → worker picks up → result delivered
5. Add `/api/job/:id` polling endpoint (free, no payment)
6. USDC payment goes directly to clawdbotatg.eth wallet on Base

## x402 SDK Details (v2.5.0, researched 2026-03-04)

### Key Packages
- `@x402/core` — core protocol, facilitator client, types
- `@x402/next` — Next.js middleware (`paymentProxy` for pages, `withX402` for API routes)
- `@x402/evm` — EVM payment scheme (exact scheme)
- `@x402/fetch` — client-side fetch wrapper with auto-payment
- `@x402/paywall` — optional built-in paywall UI for browser requests

### Server Setup Pattern
```typescript
import { withX402, x402ResourceServer } from "@x402/next";
import { HTTPFacilitatorClient } from "@x402/core/server";
import { ExactEvmScheme } from "@x402/evm/exact/server";

// Default facilitator: https://x402.org/facilitator (Coinbase-hosted, free)
const facilitatorClient = new HTTPFacilitatorClient();
const server = new x402ResourceServer(facilitatorClient)
  .register("eip155:8453", new ExactEvmScheme()); // Base mainnet

export const POST = withX402(handler, {
  accepts: {
    scheme: "exact",
    price: "$20.00",           // Human-readable — SDK converts to USDC
    network: "eip155:8453",     // Base mainnet
    payTo: "0x11ce...",         // clawdbotatg.eth receives USDC
  },
  description: "Quick Consultation",
}, server);
```

### Key Behaviors
- `withX402` settles payment ONLY after handler returns success (status < 400) — client never charged for failed requests
- Browser requests get a paywall UI, API clients get JSON 402 with `PAYMENT-REQUIRED` header
- v2 protocol: payment in `PAYMENT-SIGNATURE` header (base64-encoded)
- Facilitator verifies + settles the payment on-chain (USDC transfer on Base)
- No API keys needed for the default facilitator

### Client Usage (how agents will call us)
```typescript
import { wrapFetchWithPaymentFromConfig } from "@x402/fetch";
import { ExactEvmScheme } from "@x402/evm";
import { privateKeyToAccount } from "viem/accounts";

const account = privateKeyToAccount("0x...");
const fetchWithPayment = wrapFetchWithPaymentFromConfig(fetch, {
  schemes: [{ network: "eip155:8453", client: new ExactEvmScheme(account) }],
});

// Automatically handles 402 → pay → retry
const response = await fetchWithPayment("https://services.clawdbotatg.eth.link/api/consult/quick", {
  method: "POST",
  body: JSON.stringify({ description: "I want to build a token dashboard" }),
});
```

## Open Questions

1. **Auto-accept:** Should x402 jobs auto-assign to a worker, or go through the normal OPEN → accept flow? (Recommendation: auto-assign since pre-paid)
2. **Payment destination:** USDC goes to clawdbotatg.eth directly via x402 (off-contract). Optionally also post job on-chain for transparency.
3. **Async vs sync:** Consultations could be:
   - **Sync:** Agent waits, bot processes, returns result in same HTTP response (with long timeout)
   - **Async:** Return job ID immediately, agent polls for result
   - Recommendation: **Async** with polling endpoint
4. **A2A protocol:** Should we also add an A2A (Agent-to-Agent) endpoint? The 8004 spec supports it.
